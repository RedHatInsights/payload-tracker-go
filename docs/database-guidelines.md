# Database Guidelines

## Models

- Two model packages exist: `internal/models/models.go` (used by API queries) and `internal/models/db/models.go` (used by the Kafka consumer and write operations). Keep both in sync when changing schema-related struct fields.
- Use GORM struct tags to declare constraints (`gorm:"primaryKey;not null;autoIncrement;type:bigint"`). Prefer `type:varchar` for string columns; avoid fixed-length `char` types.
- `PayloadStatuses` has a composite primary key: `(id, date)`. The `date` field is required for partition routing. Include `gorm:"primaryKey"` on both `ID` and `Date`.
- Lookup tables (`Services`, `Sources`, `Statuses`) use `int32` IDs with unique name constraints. Use these ID types consistently when adding foreign key references.

## Migrations

- Use `golang-migrate/migrate/v4` with SQL files in `migrations/`. File naming follows the pattern `000NNN_description.up.sql` / `000NNN_description.down.sql`.
- Run migrations via the `pt-migration` binary (`internal/migration/main.go`): `./pt-migration upgrade` or `./pt-migration downgrade`. The `downgrade` command steps back one migration (`m.Steps(-1)`).
- The migration init container runs `./pt-migration upgrade` before the API pod starts (defined in `deployments/clowdapp.yml` under `initContainers`).
- Include both `.up.sql` and `.down.sql` files for every migration. The down migration should reverse changes in dependency order (drop functions before tables that use them).

## Partitioning

- `payload_statuses` is range-partitioned by the `date` column (`PARTITION BY RANGE (date)`). Each partition covers exactly one day.
- Partition names follow the format `partition_YYYYMMDD_YYYYMMDD` (generated by the `get_date_string` PL/pgSQL function).
- The `create_partition(start, stop)` function creates a partition and seven per-partition indexes: a unique `id` index plus btree indexes on `payload_id`, `service_id`, `source_id`, `status_id`, `date`, and `created_at`.
- New partitions are created by the scheduled vacuum job, not by application code. When adding queries that filter on `date`, confirm the partition for the target date range exists.

## Data Retention and Vacuum Job

- A CronJob named `vacuum` runs on schedule `CLEANER_SCHEDULE` (default `"00 17 * * *"`). The script is in `deployments/clowdapp.yml` under the `clean.sh` ConfigMap.
- The job performs three operations in order: (1) create tomorrow's partition with retry logic, (2) drop the partition for `RETENTION_DAYS` ago (default 7), (3) delete orphaned payloads older than `RETENTION_DAYS` and run `VACUUM ANALYZE payloads`.
- When changing `RETENTION_DAYS`, verify that partition creation and drop intervals stay aligned. The drop call targets a single day's partition.

## Indexing

- The parent `payload_statuses` table defines composite indexes: `(service_id, created_at)`, `(service_id, date)`, `(service_id, status_id)`, `(status_id, created_at)`, `(status_id, date)`. Each partition additionally gets its own single-column btree indexes via `create_partition`.
- `payloads` has individual btree indexes on `account`, `created_at`, `inventory_id`, `system_id`, and unique indexes on `id` and `request_id`.
- Lookup tables (`services`, `sources`, `statuses`) have unique indexes on both `id` and `name`.
- When adding a new query with a `WHERE` clause on `payload_statuses`, verify it can use existing composite indexes or add a new index inside the `create_partition` function so every partition gets it.

## Connection and GORM Configuration

- `internal/db/db.go` exposes a package-level `DB *gorm.DB` singleton initialized by `DbConnect`. The DSN is built from `config.DatabaseCfg` fields.
- SSL mode is `disable` by default; it switches to `require` when `RDSCa` is set (Clowder-managed RDS environments).
- GORM is initialized with default `gorm.Config{}` (no custom logger, no special settings). Pass `*gorm.DB` to query functions; do not create additional connections.
- A separate `DbSqlConnect` function returns a raw `*sql.DB` for use by `golang-migrate` only.

## Query Patterns

- API read queries (`internal/queries/queries_api.go`) accept a `*gorm.DB` and chain `.Where()`, `.Order()`, `.Limit()`, `.Offset()` calls. Use parameterized `?` placeholders in `.Where()` calls to prevent SQL injection.
- For cross-table queries on `payload_statuses`, use `.Table("payload_statuses").Select(fields).Joins(...)` with explicit JOIN clauses. Sources use `FULL OUTER JOIN` (nullable FK); other lookups use `JOIN`.
- Consumer write queries (`internal/queries/queries_consumer.go`) use `clause.OnConflict` for upserts on `payloads` (conflict on `request_id`). Build the `DoUpdates` column list dynamically, including only non-empty fields.
- When inserting into `PayloadStatuses` with no source, use `.Omit("source_id")` to avoid writing a zero-value FK.
- The consumer retries `InsertPayloadStatus` up to `DBRetries` times (default 3) on failure.

## Caching

- Lookup values (statuses, services, sources) are cached in-memory using `hashicorp/golang-lru/v2/expirable` with a 12-hour TTL. The cache implementation is selected via `consumer.payload.fields.repo.impl` config (`"db"` or `"db_with_cache"`; default is `"db_with_cache"`).
- The `PayloadFieldsRepository` interface wraps `GetStatus`, `GetService`, `GetSource`. Add new lookup caching by implementing this interface pattern with `PayloadFieldsRepositoryFromCache`.

## Testing

- Database integration tests use the `test.WithDatabase()` helper (`internal/utils/test/fixtures.go`), which opens a fresh GORM connection per test using `BeforeEach` / `AfterEach` (Ginkgo lifecycle).
- Tests run against a real PostgreSQL instance configured by the same `config.Get()` environment variables. Run `docker compose up payload-tracker-db` and `make run-migration` before running tests.
- Test functions for queries live in `internal/queries/queries_test.go`; endpoint integration tests with real DB are in `internal/endpoints/endpoints_db_test/`.
- Seed data for manual testing uses `tools/db-seeder/main.go` with `tools/db-seeder/seed.json`.

## Verification

```bash
# Run all tests (requires a running PostgreSQL instance with migrations applied)
make test

# Verify migration files are paired
ls migrations/*.up.sql | sed 's/.up.sql//' | while read f; do
  [ -f "${f}.down.sql" ] || echo "Missing down migration: ${f}.down.sql"
done

# Check that GORM model struct tags match between the two model packages
diff <(grep 'gorm:"' internal/models/models.go | sed 's/^[[:space:]]*//' | sort) \
     <(grep 'gorm:"' internal/models/db/models.go | sed 's/^[[:space:]]*//' | sort)

# Verify no raw string interpolation in WHERE clauses with user data (parameterized queries use ?)
# Note: Safe false positives exist where column names (not user data) are interpolated
grep -rn '\.Where(fmt.Sprintf' internal/queries/ --include="*.go" || true

# Build migration binary to confirm it compiles
go build -o /dev/null internal/migration/main.go

# Lint Go source
make lint
```
